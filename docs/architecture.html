电子题库 MVP 架构文档

本文说明这个电子题库 demo 是怎样组织起来的：哪些部分面向学生，哪些部分面向内容生产，哪些部分为未来平台化预留接口。

1. 产品边界

当前版本服务于一个目标：把教辅 PDF 中的题目转成可练习、可看解析、可形成初步学情诊断的电子题库。它不是正式教学平台，不包含账号、班级、后台编辑器和数据库。

面向学生和老师的主页面只显示题库、做题、解析和诊断；PDF 抽取、答案生成、质量标记、算法假设等技术说明全部放在文档中。

这样划分的原因是：学生使用时不需要知道题目从哪里抽取、解析是否由 agent 生成、算法内部怎么计算；老师和开发者则需要这些信息来判断内容质量、迭代 pipeline 和修正诊断逻辑。

用户看到什么不直接展示什么
学生题目、提交按钮、解析、学情诊断和推荐练习。OCR 风险、质量标记、字段 schema、pipeline 细节。
老师题库练习体验、诊断结果、文档入口。代码实现细节默认折叠在 docs 中。
开发者完整 docs、scripts、skills 和数据结构。无隐藏信息，所有规则都在仓库中。

2. 模块划分

模块职责当前实现
题库数据层保存题目、答案、解析、标签、诊断权重和推荐关系。data/questions.js
题库 UI展示题目列表、完成状态、搜索和进入做题。index.html + js/app.js
练习 UI显示题图、选项、提交、重做、参考解析。js/app.js
诊断引擎根据作答记录计算知识点、能力证据、错因和推荐。浏览器端函数 computeDiagnosis()
文档中心解释架构、pipeline、算法和 schema。docs/*.html

这些模块之间的核心约束是：题目数据层只提供结构化事实，UI 只负责展示和交互，诊断引擎只消费作答记录和题目诊断字段。这样未来更换 PDF 抽取方式、改诊断算法或接入后端时，不需要重写整个页面。

3. 数据流

PDF 页面图 -> 题目截图 -> 结构化题目数据 -> 静态网页渲染 -> 本地作答记录 -> 学情诊断 -> 推荐题

题目数据是当前 demo 的中心接口。网页不直接读取 PDF，也不在运行时调用模型生成解析。所有内容都在构建阶段生成并写入 questions.js。

运行时的数据流更具体地说是：

  1. 浏览器打开 index.html。
  2. 页面加载 data/questions.js，获得 20 道题的结构化数据。
  3. 题库页按题目数据渲染列表。
  4. 学生进入某题，页面显示题目截图、选项或解答输入框。
  5. 学生提交后，系统把作答结果写入 localStorage。
  6. 学情诊断页读取 localStorage 和 questions.js，计算知识点、能力、错因和推荐。

这意味着 demo 没有服务端依赖。只要浏览器能加载静态文件，就可以使用。

4. 状态管理

作答记录保存在浏览器 localStorage 中，键名为 pdfQuestionBankAttempts。记录包含选择项、是否正确和提交时间。清空记录只影响当前浏览器。

选择 localStorage 是为了让 demo 可以直接部署到 GitHub Pages，不需要数据库。它的代价是：不同设备之间不会同步，换浏览器也不会保留记录。

一次选择题作答记录大致长这样：

{
  "q001": {
    "values": ["C"],
    "correct": true,
    "submittedAt": "2026-06-22T06:00:00.000Z"
  }
}

一次解答题记录会多一个自评结果。当前 demo 不做自动批改，学生需要点击“我做对了”或“需要复习”。

5. 静态站为什么可以承载这个 demo

这个项目的页面交互不依赖后端，因为所有题目、解析和诊断权重都已经写在前端数据里。GitHub Pages 只需要负责把静态文件发给浏览器。

能力静态站能否支持原因
看题和做题可以题目数据和图片都在仓库内。
本地学情诊断可以诊断只需要当前浏览器的作答记录。
跨设备同步不可以需要账号和后端数据库。
老师后台编辑不可以需要权限、表单、审核和持久化存储。

6. 可升级路径

从静态 demo 升级到平台时，保持题目 schema 和前端渲染组件稳定，把本地文件读取替换为 API，把 localStorage 替换为服务端作答记录。这样可以保留题库 UI、练习 UI 和诊断逻辑的主要结构。

推荐的演进顺序是：

  1. 先扩大题库数据量，保持当前 schema 稳定。
  2. 增加题目审核和质量状态，确保答案解析可靠。
  3. 接入账号系统，把作答记录从 localStorage 移到数据库。
  4. 积累真实作答矩阵后，再升级诊断算法。
  5. 最后建设老师后台和批量 PDF 导入流程。

这个顺序可以避免一开始就做一个很重的平台，却没有足够高质量题目和诊断数据支撑。
