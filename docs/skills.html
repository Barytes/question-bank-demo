项目内 Skills 实现原理

本文说明当前项目中的三个 skill 是什么、为什么要这样拆、每个 skill 负责什么，以及它们如何协作生成题库 demo。

1. 这里的 skill 是什么

本项目中的 skill 不是安装到全局环境里的 Codex skill，而是放在 skills/ 目录下的项目内工作流说明。它们的作用类似“给 agent 和开发者看的操作手册”。

之所以把流程写成 skill，是因为 PDF 题库生产不是一次性脚本，而是一组可重复的工作：

  1. 从 PDF 中抽取题目。
  2. 生成答案、解析和标签。
  3. 根据作答记录生成学情诊断。

这三步边界不同、质量要求不同、可自动化程度也不同，所以拆成三个 skill。

2. 三个 skill 的分工

Skill输入输出核心责任
pdf-extraction扫描版 PDF页面图、题目截图、来源页把不可直接编辑的 PDF 转成可核验题目资产。
solution-generation题目截图和结构化草稿答案、解析、标签、质量标记把题目变成可练习、可解释、可诊断的数据。
learning-diagnosis题库数据和作答记录知识点掌握、能力证据、错因、推荐把做题结果转成学习反馈。

3. pdf-extraction 的实现原理

pdf-extraction 解决的是“题目从哪里来”的问题。扫描版 PDF 没有可靠文本层，所以不能简单用文本抽取。

当前实现采用图像路线：

  1. 用 pdftoppm 把 PDF 页面渲染成 PNG。
  2. 从页面图中选择完整题区。
  3. 用 Pillow 按坐标裁剪题目截图。
  4. 把截图保存到 assets/questions/。
  5. 在题目数据中记录来源 PDF、页码和截图路径。

它的关键原则是：先保留视觉证据。即使 OCR 或解析暂时不完美，老师仍然能看到原题截图，确认题目内容。

4. solution-generation 的实现原理

solution-generation 解决的是“题目如何变成可学习内容”的问题。它不只生成答案，还要生成能被页面和诊断算法使用的结构化信息。

每道题至少需要这些内容：

  • 正确答案或参考解答。
  • 解析摘要和分步过程。
  • 知识点、方法、认知能力、易错点。
  • 诊断字段：Q-matrix、能力载荷、错因证据。
  • 推荐关系：相似题、补救题、进阶题。
  • 质量标记：是否 OCR 不确定、是否需要老师核验。

这里最重要的是“显式标记不确定性”。如果答案解析来自 agent 推理，而不是已经和答案页完全匹配，就必须保留 answer_generated 和 needs_teacher_review。

5. learning-diagnosis 的实现原理

learning-diagnosis 解决的是“做完题后怎么反馈”的问题。它读取两类信息：

  1. 题目中预先写好的诊断字段。
  2. 学生保存在浏览器 localStorage 中的作答记录。

然后它按三条线计算：

计算线索使用字段结果
知识点diagnosis.qMatrix掌握概率和证据量。
能力diagnosis.abilityLoadings正负能力证据和证据充分度。
错因diagnosis.mistakeEvidence主要错因统计和补救方向。

当前模型故意保持简单，优先保证老师能看懂原因。正式平台可以在保留这些字段的基础上升级成 G-DINA、MIRT 或 BKT。

6. 三个 skill 如何串起来

pdf-extraction -> solution-generation -> learning-diagnosis

更具体地说：

  1. pdf-extraction 生成题目截图和来源信息。
  2. solution-generation 为这些题目补上答案、解析、标签和诊断字段。
  3. learning-diagnosis 在学生做题后消费这些字段，生成诊断报告。

这三个 skill 对应题库系统的三个能力层：内容生产、内容理解、学习反馈。

7. 为什么不把三个 skill 合成一个

合成一个大流程看起来更简单，但会让边界混乱。例如 PDF 裁切失败和诊断算法不合理是两类完全不同的问题；答案解析质量和页面显示也应该分开处理。

拆开之后，每个环节都可以单独改进：

  • 以后 OCR 变强，只需要替换 pdf-extraction。
  • 以后答案页对齐更准，只需要增强 solution-generation。
  • 以后有真实学生数据，只需要升级 learning-diagnosis。

这种拆法能让 demo 逐步演进成真正的平台，而不是一次性页面。
