题库数据 Schema 文档

本文定义每道题在系统里的数据结构。它既要能驱动页面展示，也要能支撑判题、解析、学情诊断和推荐。

1. Schema 的设计目标

题库数据不是简单的题干和答案。为了让系统能诊断学生，题目还需要包含知识点、能力、错因和推荐关系。

这个 schema 有四个目标：

  1. 题目能被页面直接渲染出来。
  2. 答案能被系统自动判定或用于自评。
  3. 解析能按步骤展示，方便学生复盘。
  4. 诊断字段能被算法消费，生成学情报告。

2. 顶层字段

字段说明
id稳定题目 ID，例如 q001。
title题目名称，用于列表和诊断推荐。
sourcePDF 文件名、页码和抽取方式。
questionImage原题截图路径。
typesingle、multi 或 free-response。
options选择题选项。当前显示以原题截图为准。
answer答案数组，多选题可包含多个选项。
analysis答案摘要、分步解析和选项判断。
tags知识点、方法、认知能力、易错点。
diagnosis诊断权重：Q-matrix、能力载荷、错因证据。
recommendations相似题、补救题和进阶题 ID。
qualityFlags抽取和解答质量标记。

其中最重要的接口是 answer、analysis、tags、diagnosis 和 recommendations。它们决定了系统能否从“普通题库”变成“能做学情诊断的题库”。

3. 示例

{
  "id": "q001",
  "title": "拔河中的相互作用力与静摩擦力",
  "type": "single",
  "source": { "pdf": "...pdf", "page": 3 },
  "questionImage": "assets/questions/q001.png",
  "answer": ["C"],
  "tags": {
    "knowledge": ["相互作用力", "静摩擦力"],
    "methods": ["隔离法"],
    "cognition": ["概念辨析"],
    "mistakes": ["把相互作用力大小误判为由质量决定"]
  }
}

4. 字段如何被页面使用

页面使用的字段用途
题库页id、title、type、difficulty展示题目卡片、题型和完成状态。
做题页questionImage、options、answer显示原题、提供选项、提交后判题。
解析区analysis.summary、analysis.steps显示答案概括和分步解析。
诊断页diagnosis、tags、recommendations生成掌握概率、能力证据、错因和补练推荐。
文档和维护source、qualityFlags追踪题目来源和质量状态。

5. tags 的分层

tags 被分成四类，是为了避免把不同性质的信息混在一起。

字段含义例子
knowledge课程知识点。相互作用力、静摩擦力、圆周运动。
methods解题方法。隔离法、能量法、图像法。
cognition跨题认知能力。概念辨析、模型迁移、图形表征。
mistakes常见错误模式。把速度大小相同误认为速度变化量为零。

这种分层让系统可以分别回答四个问题：学生不会哪个知识点、缺哪种方法、哪种能力证据不足、最可能犯什么错。

6. diagnosis 的结构

diagnosis 是算法真正消费的字段。

"diagnosis": {
  "qMatrix": {
    "相互作用力": 1.0,
    "静摩擦力": 0.6
  },
  "abilityLoadings": {
    "概念辨析": 0.85,
    "研究对象切换": 0.55
  },
  "mistakeEvidence": {
    "把相互作用力大小误判为由质量决定": {
      "ifWrong": 0.85,
      "ifCorrect": 0.15,
      "feedback": "若本题出错，优先检查..."
    }
  }
}

qMatrix 负责知识点掌握，abilityLoadings 负责能力证据，mistakeEvidence 负责错因诊断。更完整的解释见算法文档。

7. recommendations 的结构

推荐关系提前写入数据中，当前 demo 不做复杂实时搜索。

"recommendations": {
  "similar": ["q002"],
  "remediation": ["q003"],
  "advanced": ["q004"]
}
类型用途
similar同知识点或同题型巩固题。
remediation针对错因的补救题。
advanced更综合或更迁移的提升题。

8. 设计原则

  • 题目内容和页面渲染解耦。
  • 知识点与认知能力分开建模。
  • 原 PDF 截图作为可核验证据保留。
  • 质量不确定性必须显式记录，而不是隐藏在页面实现中。
  • 正式平台化时，尽量保持字段语义不变，只把数据来源从静态文件换成 API。

9. 升级到后端 API 时的形态

未来如果接入后端，静态 questions.js 可以拆成接口：

GET /api/questions
GET /api/questions/:id
POST /api/attempts
GET /api/diagnosis/me

只要接口返回的数据仍遵守这个 schema，前端页面和诊断逻辑就可以平滑迁移。
